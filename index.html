    agni

    Simple and intuitive MVC web framework for node.js.

    Overview

    Agni is a web framework built on top of Express, based on the principle of convention over configuration.

    Its purpose is making access to the various components easier, so that you can forget about routes and paths and focus on what your application does.

    It does not include any database abstraction system, but leaves this choice to the developer.

    Quick start

    1. Install agni globally: npm install agni -g.

    2. Create an application called welcome: agni create welcome

    3. Enter the newly created directory: cd welcome.

    4. Install all dependencies: npm install.

    5. Run the application: npm start.

    6. Point your browser to localhost:3000.

    7. If you see the welcome message, it works!

    Architecture

    Agni is loosely based on the MVC development pattern.

    The three main application components are:

    • the Model, which is the data access layer;

    • the View, which is the part of the application that is "visible" to the user;

    • the Controller, which acts as an intermediary between the other two components, taking data from the Model and sending it to the View.

    A fourth component is also available, that is the Service layer. The purpose of this layer is encapsulating business logic in a high level interface available to the controller, freeing the latter from overload and duplication.

    Of course, you are free to use only the components that you need. Agni does not oblige you to use Models and Services if you do not need them.

    Directory structure

    The root application directory is app.

    Inside app, you have the following directories:

    • controllers, models, services and views, where you can store the modules belonging to the corresponding components.

    • config, which contains three files: config.js, with the application settings, init.js, where you can define middleware, and server.js, where you can set whether to run the server as a daemon (with forever) or not.

    • public, where you can put all your static resources like images, javascript files, style sheet and so on.

    File extensions

    You can use the extensions .controller.js, .model.js and .service.js for more readable file names.

    Routing

    There is no explicit route definition in Agni. The routing mechanism is based on controller paths, function names and (optionally) HTTP methods.

    Controllers

    The Controller as a router

    Agni uses a common routing system, similar to what is used in CodeIgniter and other popular frameworks.

    If you split the requested URL in segments:

    mywebsite.com/module/function/arg1/arg2

    1. The first URL segment is the module that should be required.

    2. The second segment is the function that should be called.

    3. The third, and any other segments, are the arguments that should be passed to the function.

    So, for example, a request to mywebsite.com/article/show/173, would require the article module from the controllers folder, and execute the exported function show with 173 as parameter.

    More or less equivalent to doing:

    var article = require('./app/controllers/article');
article.show(173);

    If you omit the function name, it will look for a function called nil and execute it.

    For example, a request to mywebsite.com/articles will require the articles module and execute the nil function.

    You can create namespaces, by simply putting your controllers in sub-directories.

    For example, a request to mywebsite.com/blog/article/show/173, would require the article module from the controllers/blog folder, and execute the exported function show with 173 as parameter.

    Controller properties

    Inside controllers, you can access the standard request and response objects as this.req as this.res, respectively.

    For convenience, the following properties are also provided:

    • this.COOKIES (cookies)

    • this.SESS (cookie-based session)

    • this.GET (query string)

    • this.POST (request body)

    Resourceful routing

    You can easily map requests to CRUD operations, by setting the REST property in the controller.

    Let's suppose we have a controller called article, which looks like this:

    //File: app/controllers/article.js

exports.REST = true;

//GET
exports.read = function(id) {
  //Read
}

//POST
exports.create = function() {
  //Create
}

//PUT
exports.update = function(id) {
  //Update
}

//DELETE
exports.del = function(id) {
  //Delete
}

//GET edit
exports.edit = function(id) {
  //Edit
}

    The requests will be mapped according to the HTTP method and the URI segments:

    • GET /article/1 => article.read(1)

    • POST /article => article.create()

    • PUT /article/1 => article.update(1)

    • DELETE /article/1 => article.del(1)

    • GET /article/edit/1 => article.edit(1)

    Views

    Rendering a view

    Rendering views is very easy. Inside a controller, execute:

    this.render(viewName, locals, callback).

    If you omit the view name, it will be the same as your controller module. For example, let's suppose you have a controller module called hello:

    //File: app/controllers/hello.js

exports.nil = function() {
  //This will render the view 'app/views/hello.html'
  this.render();
}

    It will render the view hello.html.

    Models

    To access a model, you can use this.model(modelName) from any controller or service function. It will return the same object as require(__dirname + '../models/' + modelName).

    If you omit the name, it will be the same as your controller module.

    Example:

    //File: app/models/article.js

exports.readArticle = function(id, callback) {
  //Read article from DB
  //...

  callback(err, articleData);
}

    And in the controller:

    //File: app/controllers/article.js

exports.read = function(id) {
  var self = this;

  this.model().readArticle(id, function(err, data) {
    if(! err) {
      self.render('read_article', {articleData: data});
    }
  });
}

    Services

    To access a service, you can use this.service(serviceName) from any controller or service function. It will return the same object as require(__dirname + '../services/' + serviceName).

    If you omit the name, it will be the same as your controller module.

    Example:

    //File: app/services/string_helpers.js

exports.reverse = function(str) {
  return str.split('').reverse().join('');
}

    In the controller:

    exports.nil = function() {
  var str = 'Hello';
  var reversed = this.service('string_helpers').reverse(str);
  this.render({reversedHello: reversed});
}

    Service constructor

    If the service returns a constructor function, you can use the construct() convenience method:

    //File: app/services/circle.js

function Circle(radius) {
  this.radius = radius;
}

Circle.prototype.getArea = function() {
  return (Math.PI * this.radius * this.radius);
}

module.exports = Circle;

    In the controller:

    exports.nil = function() {
  var area = this.service('circle').construct(3).getArea();
}

    which is equivalent to:

    exports.nil = function() {
  var Circle = this.service('circle');
  var circle = new Circle(3);
  var area = circle.getArea();
}

    Configuration

    In all controller, model and service functions you can access the Express app.settings object as this.settings.

    If you need the settings object in the global scope of a module, you can use the application instance which is exported by /lib/agni.js:

    //File: app/models/foo.js

var app = require(__dirname + '../../../lib/agni');
var env = app.settings.env;

    Error handling

    Because Agni router is regular Connect middleware, you can pass errors to the next function, available inside controller functions as this.next(). This will trigger a 500 Internal Server Error and send the error message to stderr. HTML error pages are in the views/errors directory.

    Example:

    //File: app/controllers/article.js

exports.REST = true;

exports.read = function(id) {
  var self = this;

  this.model().findArticleById(id, function(err, result) {
    if(! err) {
      self.res.json(result);
    } else {
      self.next(err);
    }
  });
}

    Examples

    Hello world!

    Controller

    //File: app/controllers/hello.js

exports.nil = function() {
  this.render();
}

    View

    <!-- File: app/views/hello.html -->

<p>Hello world!</p>

    Open the page localhost:3000/hello.

    That's it!

    What's your name?

    Controller

    //File: app/controllers/hello.js

exports.myNameIs = function(name) {
  var locals = {name: name}
  this.render('my_name_is', locals);
}

    View

    <!-- File: app/views/my_name_is.html -->
<!-- Example with template engine ejs -->

<p>Hello, <%= name %>.</p>
<p>Nice to meet you!</p>

    Now point your browser to localhost:3000/hello/my-name-is/John, and see the output.

    Please note how function names are automatically converted to camelCase.

    Sorry if your name is not John :)
